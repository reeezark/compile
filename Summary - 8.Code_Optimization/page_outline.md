## p4-p6 优化目标、优化发生位置与原则

- 优化器由控制流分析、数据流分析、代码变换组成。
- 优化目标：提高执行时间、内存使用、能耗、二进制大小等平均表现。
- 优化发生位置：
  - 人工源码层面。
  - 编译器 IR 层面。
  - 编译器目标代码生成层面。
  - 汇编器/链接器层面，例如链接时优化。
- 三个基本原则：
  - 等价性：优化后程序结果不能改变。
  - 高效性：目标代码运行更快或占更少空间。
  - 成本效益：优化本身开销应合理。

## p7-p14 优化分类：布局相关与代码相关

- 优化本质是程序变换：删除、添加、移动、修改。
- Layout-related transformations：优化代码/数据在内存中的放置，目标是最大化空间局部性。
- 代码布局示例：函数访问序列 `g,f,h,f,h,f,h` 下，不同函数排列导致 cache miss 从 6 次降到 2 次。
- 数据布局示例：
  - 调整结构体字段顺序，使一起访问的 `a` 和 `c` 落在同一 cache line。
  - AOS 转 SOA，提高同字段连续访问的空间局部性。
- Code-related transformations：优化生成什么代码，目标是执行最少、代价最低的指令。

## p15 代码相关优化的四类变换

- Modifying：强度削减，如 `2*a` 改为 `a << 1`。
- Deleting：死代码删除，如 `A=2; A=y;` 删除前一次赋值。
- Moving：代码调度，如把独立指令提前以暴露并行性。
- Inserting：数据预取，如链表遍历中提前 `prefetch(p->next)`。

## p16-p22 控制流分析、基本块与 CFG

- 控制流分析用于确定程序执行路径，通常用控制流图 CFG 表示。
- 基本块：直线代码片段，只有第一条能进入，只有最后一条能跳出；块内指令要么全执行，要么都不执行。
- 基本块是 CFG 节点；局部优化限制在一个基本块内，全局优化跨基本块。
- CFG：节点为基本块，边表示执行流；if、while、for 等控制语句引入边。
- 构造 CFG：
  - 找 leaders：第一条指令、跳转目标、紧跟跳转后的指令。
  - 每个 leader 到下一个 leader 前形成基本块。
  - 根据 fall-through 和 jump 添加边。
- p22 示例：输入读数、判断奇偶、打印 even/odd、goto/end，leaders 为 0、5、2、4。

## p23-p33 局部优化与 DAG

- 局部优化只在一个基本块内进行，不考虑控制流。
- 常见局部优化：
  - 常量折叠。
  - 公共子表达式消除 CSE。
  - 死代码删除 DCE。
- 全局优化跨基本块，通常在函数内进行，不一定跨整个程序。
- DAG 表示基本块内部值流，便于识别公共子表达式和死代码。
- DAG 构造：
  - 为初始变量值建叶子节点。
  - 为每条语句建操作节点，其子节点是操作数最近定义。
  - 输出节点表示块出口仍需保留的值。
- DAG 优化示例：
  - 若 `b` 在块出口不活跃，可删除 `b=a-d`。
  - 若 `b` 和 `d` 都活跃，重复的 `a-d` 可只算一次，再复制。
  - 若只有 `a` 活跃，可删掉对应死代码节点。
- 代数恒等式可发现普通 DAG/CSE 不能直接发现的等价表达式。
- 强度削减：用加法、减法、移位替代乘法/除法等昂贵操作。
- 常量折叠：编译期计算常量表达式，如 `2*LEN`。
- 常量传播：把已知常量代入后续使用；局部版本简单，全局版本更强但更复杂。

## p34-p41 全局优化、保守性与数据流分析

- 全局优化扩展到 CFG 的控制流。
- 优化必须对所有路径正确；只要存在一条路径可能不满足条件，就不能做该优化。
- 示例：
  - `X=3; if(...) Y=...; Y=0; A=2*X;` 中 `X=3` 未被改写，可传播到 `A=2*3`。
  - 若 if 分支中有 `X=Y`，则不能把后面的 `X` 简单替换为 3。
- 保守性原则：
  - 只有某性质在某点一定为真才启用优化。
  - 不确定时回答 don't know，不做优化。
  - 可能错失机会，但保证正确性。
- 数据流分析：为程序每个点计算属性/值，沿 CFG 边传播，并用保守近似保证所有路径正确。
- GCP 状态：
  - `x=*`：未赋值/默认。
  - `x=1, x=2, ...`：常量值。
  - `x=#`：可能有多个值，无法确定常量。
- 迭代直到固定点，再替换可确定常量。
- GCP 后可继续 DCE，删除不再有用的赋值。

## p42-p45 LLVM Pass 与优化等级

- LLVM 优化以 Pass 形式实现。
- Pass 遍历某部分程序，收集信息或变换程序。
- Pass 输入 LLVM IR，执行分析和/或变换。
- Pass Manager 组织编译流水线中的 Pass 顺序。
- 优化等级：
  - `O0`：不优化，编译最快，最利于调试。
  - `O1`：介于 O0 与 O2。
  - `O2`：中等优化，启用多数优化。
  - `O3`：比 O2 更激进，可能更耗时或增大代码体积以换取速度。
  - `Os`：类似 O2，但额外减少代码大小。
  - `Oz`：比 Os 更进一步减小代码体积。
  - `O4`：启用链接时优化；Clang 支持 O4，opt 不支持。

## p48-p50 DAG Construction Revisit

- DAG 构造三种语句：
  - `x = y op z`
  - `x = op y`
  - `x = y`
- 构造规则：
  - 操作数未定义时创建叶子节点。
  - 二元操作创建 OP 节点，左右孩子分别为 y、z 节点。
  - 一元操作尝试复用已有同 OP 和同 child 节点。
  - 复制语句让 `x` 附着到 `y` 的节点。
  - 每次重新定义 x 时，先从旧节点标识符列表移除 x，再附到新节点。
- 示例：
  - `x=A+B`
  - `y=x+3`
  - `z=x*y`
  - `w=x*y`
  - `u=w`
  - `a=4/2`
  - `b=-a`
- 若 `z` 和 `b` 活跃，可优化为：
  - `x=A+B`
  - `y=x+3`
  - `z=x*y`
  - `b=-2`
